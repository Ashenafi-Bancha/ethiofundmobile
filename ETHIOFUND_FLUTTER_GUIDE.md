# EthioFund Flutter Guide

## Overview

EthioFund is a Flutter crowdfunding app for donors, organizers, and admins. The app is the mobile client, so Flutter handles the UI, navigation, validation, local state, and mock data. The real backend is expected to be Node.js + Express + PostgreSQL, but the app can also run fully in mock mode for UI testing.

## How The App Works

The app starts on a branded splash screen, then goes to onboarding if needed, and finally lands on the public home page. Users do not get forced straight into login. Instead, they can browse first, then choose a role and sign in or sign up when ready.

Riverpod provides state management. GoRouter controls navigation and role-based access. Dio handles HTTP calls when the real backend is enabled. Secure storage keeps auth tokens and user data. Shared preferences stores the onboarding flag. Mock mode uses the same service layer, but reads from in-memory fake data instead of the network.

## Main Features

- Onboarding and first-run detection
- Public home page with role choice for donor, organizer, and admin
- Login and registration with a clean, simple layout
- Campaign browsing and detail pages
- Donation checkout handoff and payment return handling
- Comments and organizer updates
- Campaign creation and editing
- Withdrawal request submission and history
- Profile viewing, editing, and sign out
- Admin dashboard, users, campaigns, withdrawals, and reports
- Mock data for full UI testing without a backend

## Folder Structure

### What is in this project

This repository contains a Flutter mobile app only. The important parts are:

- `lib/`: all application code.
- `assets/`: images and other static resources.
- `test/`: widget tests and smoke tests.
- `android/`, `ios/`, `web/`, `windows/`, `linux/`, `macos/`: platform folders generated by Flutter.
- `build/`: generated build output.

### `lib/main.dart`

App entrypoint. It initializes Flutter bindings, locks orientation to portrait, seeds the mock onboarding state when mock mode is enabled, wraps the app in `ProviderScope`, and launches `EthioFundApp`.

### `lib/app.dart`

Top-level app widget. It builds `MaterialApp.router`, applies the EthioFund theme, and connects the router.

### `lib/core/`

Shared infrastructure used across the app.

- `constants/`
  - `api_constants.dart`: backend routes and base URL.
  - `app_environment.dart`: switches between mock mode and real backend mode.
  - `app_colors.dart`: brand colors.
- `network/`
  - `dio_client.dart`: configured Dio instance with JWT injection.
  - `api_exception.dart`: friendly error mapping.
- `router/`
  - `app_router.dart`: all routes, redirects, role checks, and shell navigation.
- `storage/`
  - `secure_storage.dart`: JWT and user storage.
  - `app_preferences.dart`: onboarding flag stored with shared preferences.
- `theme/`
  - `app_theme.dart`: Material 3 theme for the app.
- `utils/`
  - `formatters.dart`: date, progress, and currency helpers.
  - `validators.dart`: form validation helpers.

### `lib/mocks/`

Fake backend data used when `AppEnvironment.useMockData` is `true`.

- `mock_backend_store.dart`: seeded users, campaigns, comments, donations, withdrawals, and reports.

### `lib/shared/widgets/`

Reusable UI pieces used across screens.

- `brand_mark.dart`: small logo widget used in the app bar, splash, and navigation.
- `auth_header.dart`: logo + title + subtitle block for login and sign-up.
- `app_shell.dart`: bottom navigation and wide-screen navigation rail.
- `primary_button.dart`, `status_badge.dart`, loading/error widgets, and other common components.

### `lib/models/`

Typed API models.

- `user_model.dart`: signed-in user data.
- `campaign_model.dart`: campaign data.
- `campaign_update_model.dart`: organizer updates.
- `donation_model.dart`: donation history and donation details.
- `comment_model.dart`: campaign comments.
- `withdrawal_model.dart`: withdrawal requests.
- `dashboard_stats_model.dart`: admin dashboard summary.

### `lib/services/`

Service classes that talk to either the mock store or the real backend.

- `auth_service.dart`: login, register, logout.
- `campaign_service.dart`: campaign CRUD and updates.
- `donation_service.dart`: donation history.
- `payment_service.dart`: checkout initialization.
- `comment_service.dart`: comments.
- `withdrawal_service.dart`: withdrawal requests and history.
- `user_service.dart`: profile fetch and update.
- `admin_service.dart`: admin dashboard, user moderation, campaign moderation, withdrawals, and reports.

### `lib/providers/`

Riverpod state and data providers.

- `auth_provider.dart`: auth state and login/logout actions.
- `campaign_provider.dart`: campaign list, detail, and filtered views.
- `donation_provider.dart`: donation-related lists.
- `comment_provider.dart`: campaign comments.
- `withdrawal_provider.dart`: withdrawal history.
- `profile_provider.dart`: profile fetch.
- `admin_provider.dart`: admin data streams.

### `lib/features/`

Feature modules and screens.

- `splash/`: startup routing and brand splash.
- `onboarding/`: first-run onboarding.
- `auth/`: role-aware login and register screens.
- `home/`: public home page, discovery hero, and browse screen.
- `campaign/`: campaign detail, create, edit, and organizer campaigns.
- `donation/`: donate flow, WebView payment, success/failure, and donation history.
- `withdrawal/`: withdrawal request and history.
- `profile/`: user profile screen.
- `admin/`: admin dashboard and moderation screens.

### `lib/shared/`

Reusable UI and helpers.

- `widgets/`: buttons, loading, error, empty state, badges, shell, and other shared widgets.
- `extensions/`: context helpers.

## Navigation Flow

1. `SplashScreen` shows the logo, then routes to onboarding or the public home page.
2. `OnboardingScreen` marks onboarding as seen and returns to home.
3. The home page lets users choose a role before login or sign-up.
4. `GoRouter` guards protected routes based on auth and role.
5. `AppShell` keeps the main sections alive while switching tabs.

## Backend Contract

The Flutter app expects the backend to return JSON responses that match the fields used in the models and services. If you change the backend, update `lib/core/constants/api_constants.dart` and the related model parsing. In mock mode, the same services still work, but they read from the in-memory mock store instead of HTTP.

## Running The App

```bash
flutter pub get
flutter run
```

## Mock Mode

The app now starts in mock mode so you can test the full UI without a backend.

Sample accounts:

- `admin@ethiofund.test`
- `organizer@ethiofund.test`
- `donor@ethiofund.test`

Any valid password is accepted in mock mode.

Mock mode is controlled by `lib/core/constants/app_environment.dart`. Set `AppEnvironment.useMockData` to `false` when you are ready to connect the real API.

The seeded mock data includes:

- campaigns in approved, pending, and rejected states
- comments and campaign updates
- donation history
- withdrawal requests
- admin dashboard stats, users, campaigns, withdrawals, and reports

For tests:

```bash
flutter test
```
